# Supabase Integration — Simple Password Table + Data Sync

## Goal

Replace the hardcoded `'admin'` password with a SHA-256 hash stored in a Supabase `config` table, and sync both marker data and grid settings to Supabase so the map is shared across browsers. localStorage remains as an offline fallback. No RLS — client-side password check only (same security tier as the current hardcoded string).

## Constraints

- No build step, no modules — single IIFE in `HexMap.js`, `supabase-js` loaded via CDN `<script>` tag.
- The app must work identically on `file://` with no internet. All Supabase calls guarded by `typeof supabase !== 'undefined'`.
- Single shared map, one editor, many viewers.
- No password-change UI — the hash is set via Supabase SQL editor.

### Setup SQL

Run once in the Supabase SQL editor to create both tables and seed the config row:

```sql
CREATE TABLE IF NOT EXISTS config (
    id int4 PRIMARY KEY DEFAULT 1,
    password_hash text NOT NULL DEFAULT '',
    grid_type text NOT NULL DEFAULT 'hexRowsOdd',
    grid_width int4 NOT NULL DEFAULT 40,
    grid_height int4 NOT NULL DEFAULT 40,
    grid_style text NOT NULL DEFAULT 'solid',
    grid_thickness float4 NOT NULL DEFAULT 0.5,
    grid_color text NOT NULL DEFAULT '#ffaa33',
    grid_opacity float4 NOT NULL DEFAULT 0.3,
    zoom float4 NOT NULL DEFAULT 1.0,
    grid_pan_x float4 NOT NULL DEFAULT 0,
    grid_pan_y float4 NOT NULL DEFAULT 0,
    scene_width text NOT NULL DEFAULT '6000',
    scene_height text NOT NULL DEFAULT '4000',
    offset_bg_horizontal text NOT NULL DEFAULT '0',
    offset_bg_vertical text NOT NULL DEFAULT '0',
    CONSTRAINT single_row CHECK (id = 1)
);

INSERT INTO config (id, password_hash) VALUES (1, '')
ON CONFLICT (id) DO NOTHING;

CREATE TABLE IF NOT EXISTS markers (
    id int8 GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    col int4 NOT NULL DEFAULT 0,
    row int4 NOT NULL DEFAULT 0,
    color text NOT NULL DEFAULT '#ff3b3b',
    shape text NOT NULL DEFAULT 'star',
    identifier text NOT NULL DEFAULT '',
    details text NOT NULL DEFAULT ''
);
```

### Setting the password

Generate the SHA-256 hash of your desired password and update the config row. Example using the browser console:

```js
async function hashPassword(pw) {
    const buf = await crypto.subtle.digest('SHA-256', new TextEncoder().encode(pw));
    return Array.from(new Uint8Array(buf)).map(b => b.toString(16).padStart(2, '0')).join('');
}
hashPassword('your-password-here').then(h => console.log(h));
```

Then in Supabase SQL editor:

```sql
UPDATE config SET password_hash = '<paste-hex-hash-here>' WHERE id = 1;
```

## 3. Data Flow

### Load (`init()`)

1. Call `loadFromSupabase()` — fetches `config` row and all `markers` rows in parallel.
2. **Success:** Apply fetched data to `state` and `markers`. Mirror to localStorage (so the cache is fresh). Render.
3. **Failure (network error, sb is null):** Call existing `loadState()` from localStorage. Show an offline notice (see section 5). Render.

`loadFromSupabase()` is a new async function. It returns `{ config, markers }` or throws on failure. `init()` becomes async — the load must complete before the first `render()` call so the user sees Supabase data, not a blank canvas followed by a flash of remote data.

### Save

`saveState()` is modified:

1. Write to localStorage immediately (existing behavior, unchanged).
2. If `sb` is available and `isEditMode` is true, fire-and-forget `syncToSupabase()`.

`syncToSupabase()` is a new async function:

1. Upsert the `config` row (id=1) with all current settings. Does NOT write `password_hash` — that field is read-only from the app's perspective.
2. Delete all rows from `markers`, then insert the full current `markers` array.
3. Both operations use a single call per table (Supabase's `.upsert()` and `.insert()` batch methods).
4. On failure: `console.warn()` only. No user-facing error. localStorage already has the data.

The delete-all + re-insert approach is simple and correct for a single-editor map with low marker counts. No diffing, no conflict resolution.

### Auth (`submitAuth()`)

1. If `sb` is available: fetch `config.password_hash` from Supabase. On success, cache the hash in localStorage under a new key `hexMapPasswordHash`.
2. If fetch fails: read the cached hash from `localStorage.getItem('hexMapPasswordHash')`.
3. If no hash available at all (first load, offline, never connected): show HUD alert "Connection required for first-time setup."
4. SHA-256 hash the user's input using `crypto.subtle.digest()`.
5. Compare hex strings. Match: enter edit mode. Mismatch: show error.

The existing `submitAuth()` function is modified to be `async` (it already calls `updateViewportHud` etc. synchronously after the check — those calls stay after the `await`).

## 6. Verification Checklist

1. **Online load:** Open page with Supabase reachable. Markers and settings load from Supabase. No offline notice.
2. **Offline load:** Disconnect network (or use `file://`). Page loads from localStorage. Offline notice appears, auto-dismisses.
3. **Auth (online):** Type correct password → edit mode. Type wrong password → error. Password is never visible in source (only the hash is fetched).
4. **Auth (offline, cached):** Disconnect after one successful load. Auth still works using cached hash.
5. **Auth (offline, no cache):** Clear localStorage, disconnect. Auth shows "connection required" alert.
6. **Save markers:** Add/edit/delete a marker in edit mode. Reload page → marker changes persist (both Supabase and localStorage).
7. **Save settings:** Change grid type, dimensions, zoom. Reload → settings persist.
8. **Cross-browser sync:** Edit on browser A. Open browser B → sees the same markers and settings.
9. **file:// mode:** Open HexMap.html as a local file. App works exactly as before (localStorage only, supabase-js CDN doesn't load, no errors in console).
10. **No native dialogs:** Grep `HexMap.js` for `\b(alert|prompt|confirm)\(` — zero matches outside comments.
